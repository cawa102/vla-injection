One question in one sentence: when someone secretly hijacks a robot's instructions, which kind of "inspector" is hardest for the attacker to sneak past — and how much does sneaking past it cost them?

The Big Picture

What this project is about

Modern robots can be controlled by a single AI model that looks at a camera image, reads a plain-English instruction, and decides how to move. These are called VLA models — Vision-Language-Action. This project studies one called OpenVLA-7B, running in a simulator (no real robot), on a manipulation benchmark called LIBERO.

Think of a VLA model like a very literal kitchen helper: you hand it a photo of the worktop and a sticky note that says "put the red cup on the tray", and it moves its arm to do exactly that.

The threat: someone tampers with the instruction

An attacker can secretly add a hidden chunk of text to the instruction — a carefully crafted adversarial suffix — that quietly redirects the robot's movements to a target the attacker chose. The published attack we reproduce is called RoboGCG, and it works over 90% of the time. This is an integrity problem: the robot ends up doing the wrong thing while looking like it's following orders.

It's as if someone scribbled an extra line on your sticky note in invisible ink — the helper now reaches for a different object, even though the note still looks normal to you.

What we actually do: measure, don't promise

We add "inspectors" that watch for hijacked behaviour, and we measure how good each one is — and, crucially, how much attack-success an attacker has to give up to fool it. That give-up cost is the "Embodiment Evasion Tax". The big idea: checking the robot's actual movements may be harder to fool than checking the text or the model's inner signals, because movements are tied to the laws of physics.

This is a measurement study — not a claim of a perfect defence.

The honest result could be "this inspector is hard to fool" or "it's easy to fool". Both are useful, publishable findings. We avoid over-claiming on purpose.

Research Background

What prior work has already shown

Before adding any inspector, it helps to know what's already been proven — and exactly where today's defences fall short. Our starting point is the published attack we reproduce: RoboGCG (Jones et al., 2025 — Gray Swan AI, CMU & UPenn).

The attack, in detail

RoboGCG is a white-box attack: the attacker can see the model's weights and uses an optimisation algorithm (GCG) to search for a short text suffix that gets glued onto the instruction once, at the start of the run. The goal isn't vague mischief — it's to force the model to emit one specific, attacker-chosen low-level action (one of 256⁷ possible 7-number moves). They ran it on OpenVLA-7B fine-tuned for four LIBERO simulation suites (Goal, Object, Spatial, and the harder long-horizon LIBERO-10).

An honest nuance: this is "control authority", not proven harm.

The paper shows the attacker can make the model output the exact action they picked — it does not claim the robot then completes a dangerous real-world task. That distinction is one we keep carefully throughout this project too.

Like a master key filed down by trial-and-error until it throws one exact bolt — it doesn't "pick locks in general", it hits the single target the attacker chose.

The defences they tried — and why each fell short

The same paper borrowed three defences from the LLM-jailbreak world and tested them against the attack. The result is the gap this whole project is built around: each one either doesn't work, or only works by becoming unusable.

no effect

Multimodal perplexity filter

Flags inputs that look statistically "weird" across the image and text together. Attack success was essentially unchanged.

The image embeddings dominate the score, so the odd text suffix is effectively invisible to it.

no shippable setting

Text-only perplexity filter

Reads the text alone and can drop attack success to 0% — but only if you already know the correct threshold.

That threshold "cannot be known beforehand in open-world robotics" — so there's no operating point you can actually deploy.

breaks the robot

Input smoothing / paraphrase

Scrambles and rewrites the instruction to disrupt the suffix. It also drops attack success to 0%…

…but garbles the real instruction too, so benign tasks also fall to 0% success. The cure kills the patient.

The authors' own conclusion: this is an open problem.

They argued the field needs a notion of "VLA refusal" — a way to stop a robot when its control is being subverted. A defence with a false-alarm rate you can set in advance and a measured cost to benign tasks is exactly the missing piece this project measures. (The L0 text filter above becomes our baseline to beat.)

And it isn't only robots: a 2025 study (The Attacker Moves Second, Nasr et al.) showed adaptive attackers bypass 12 recent LLM defences more than 90% of the time. That's why this project measures evasion cost rather than promising an unbreakable fix.

Under the Hood

How the pieces fit together

Here's the full pipeline, from instruction to verdict. The attacker tampers with the instruction at the very start; everything downstream tries to notice that something went wrong.

An action is just seven numbers each step: six to move the arm and one to open or close the gripper. The model outputs them as discrete "tokens", and our action_codec turns those tokens back into real movement numbers.

Three inspectors, judged fairly

We don't use one inspector — we compare three, each looking at a different place in the pipeline. The fairness rule: every inspector is tuned to the same false-alarm rate (e.g. it's allowed to wrongly stop an innocent robot only 5% of the time). Only then do we compare how often each one catches the real attack.

The Star Instrument

The L2 action monitor & the "tax" idea

The heart of the project is the L2 goal-action consistency detector. At each step it asks: "given the trusted goal, do the last few actions make sense?" If the robot is supposed to reach for the tray but its arm keeps drifting somewhere else, that's inconsistent — raise the alarm.

It's causal

It only looks at past and current actions — never the future, because a real robot can't peek ahead.

It uses a window

One action is ambiguous, so it scores a short sequence of recent actions together.

It's calibrated

You set its false-alarm budget up front (say 5%) — the missing ingredient older defences lacked.

So what is the "Evasion Tax"?

To slip past a good inspector, an attacker usually has to make their attack weaker — for example, attack a less ambitious target so the movements look more normal. The tax is exactly that loss: how much attack-success must the attacker forfeit to avoid being caught? A higher tax means a stronger inspector.

Like a shoplifter who, to avoid the camera, can only grab the items near the exit — they get away unseen, but they steal far less. The "tax" is everything they had to leave behind.

The project's bet — to be measured, not assumed — is that the L2 action monitor charges a higher tax than the L0 text filter or the L1 internal probe, because physical movements are constrained by the world in a way that text and hidden signals are not.

Honesty First

The two headlines & what we won't claim

The "tax" actually splits into two separate findings. Keeping them apart is a load-bearing rule in this project — mixing them up would be over-claiming.

H6-A · guaranteed

The oracle frontier

Using a perfect "cheating" inspector that can see the simulator's ground truth, we measure the intrinsic cost physics imposes on an ideal attacker. This is a floor result we get no matter what — but it's a best-case lab measurement, not a real deployable defence.

H6-D · the headline (if affordable)

The deployable cross-layer tax

A realistic inspector (no cheating) vs a realistic adaptive attacker, comparing L1 and L2 fairly. This is the true "tax" headline — but it's expensive, so we only commit to it if the GPU timing test says it fits the calendar.

Why the split matters.

Only H6-D earns the word "tax". If the budget can't support it, we publish H6-A honestly and report the cross-layer tax as unresolved — the title even changes to match. We never let the floor result quietly carry the bigger claim.

✅ What we DO claim

  • Goal-action consistency can detect injected actions — measured against a real attack.
  • The detector has a calibrated, set-in-advance false-alarm rate (the key novelty).
  • We report operational numbers no VLA benchmark gives: false-abort rate + confidence intervals, detection latency, benign task-success cost.
  • An idealized frontier isolates embodiment as a defensive constraint.

🚫 What we do NOT claim

  • That this is an unbreakable defence — an adaptive attacker can fight back.
  • That the "runtime VLA-safety" research lane is empty — four 2026 papers already occupy nearby ground; we cite and differentiate.
  • That it's a new defence paradigm — it's a measuring instrument.
  • Anything about real robots, physical/visual attacks, or backdoors — strictly simulation, instruction-channel only.

The compute branches (chosen by the M1 timing test)

We don't yet know how fast the attack and probes run on the granted GPU. So we pre-register three honest paths and let the measured speed pick one — no wishful thinking:

Branch N

Full tax. Everything fits → commit the H6-D headline in full, with room to spare.

Branch N−

Scoped tax. Tight budget → a smaller but still real H6-D, with the gaps reported honestly.

Branch F

Oracle only. Too slow → publish H6-A; mark the deployable tax unresolved; title falls back.

Hardware: Kelvin2 (A100/H100 GPUs, NI-HPC at QUB), running OpenVLA-7B in bf16. Reproducibility rule: log the exact card + settings every run, and never compare results across different hardware within one claim.

The inspectors' brains src/evasion_tax/metric/

state.py
A clean adapter for the simulator's "privileged" ground truth (where the arm, gripper, and objects really are).

The L1 inspector: a small classifier reading the model's hidden activations to spot an injection.
probe_confounds.py

The decision maker src/evasion_tax/detector/

decide.py
Makes the causal stop/allow call step-by-step, and measures detection latency (how late the alarm fired).

perplexity.py
The L0 text-only filter: flags instructions whose wording looks statistically odd.

Plain-English Dictionary

Glossary, with analogies

VLA model Vision-Language-Action

An AI that takes a camera image + a text instruction and outputs robot actions.

a kitchen helper that acts on a photo plus a sticky note.
OpenVLA-7B

The specific open-source VLA model we study, with 7 billion parameters.

the particular "brain" we put under the microscope.
bf16 bfloat16

A compact number format that lets the big model fit on one GPU without losing much accuracy.

packing a suitcase efficiently so everything still fits.
Instruction injection

Sneaking hidden text into the instruction to redirect the robot's behaviour.

a forged extra line on the sticky note.
GCG / RoboGCG

An algorithm that searches for an adversarial text suffix; RoboGCG is its robot version (>90% success).

a lock-pick tuned by trial-and-error until the lock opens.
LIBERO

The robot manipulation simulator and task set we run everything in — no real hardware.

a flight simulator, but for a robot arm.
Oracle

A "cheating" inspector allowed to see the simulator's hidden ground truth. Great for measuring limits; not deployable.

an exam marked by someone holding the answer key.
FPR / TPR

False-Positive Rate = how often it wrongly stops an innocent robot. True-Positive Rate = how often it catches a real attack.

a smoke alarm: don't beep at toast (FPR), do beep at fire (TPR).
Calibration

Tuning the alarm threshold in advance so the false-alarm rate equals a chosen budget — the key novelty here.

setting a thermostat to a target before the room heats up.
Pareto frontier

The curve of best possible trade-offs — here, attack-success vs. avoiding detection.

the menu of "best deals" where you can't improve one thing without sacrificing another.
Evasion tax

How much attack-success an attacker must give up to avoid being caught. Bigger = stronger inspector.

what the shoplifter leaves behind to dodge the camera.
L0 / L1 / L2

The three inspectors: text filter (L0), internal-signal probe (L1), and action monitor (L2).

checking the note, the helper's thoughts, and its actual hands.
ASR Attack Success Rate

How often the attack achieves its target.

the lock-pick's batting average.
SABER

A stealthier, fluent-sounding attack (low perplexity) that the L0 text filter struggles to catch — a possible later arm.

a forged note in perfect handwriting.
