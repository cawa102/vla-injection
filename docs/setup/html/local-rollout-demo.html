Real vs stand-ins

The point is to inspect the GPU experiment's record structure locally before GPU day-1. OpenVLA-7B is the only required stand-in because it needs CUDA/bf16.

Real experiment on GPU Local demo stand-in Why the swap exists
OpenVLA-7B emits the 7-DoF action placeholder policy: scripted reach or seeded random OpenVLA-7B needs CUDA + bf16; absent on an 8 GB Mac.
LIBERO simulates the rollout real LIBERO with --backend libero; synthetic fallback otherwise State-only ControlEnv now runs locally; no GL and no torch.
RoboGCG adversarial suffix demo/placeholder-suffix The real attack is an M1 GPU-node step; demo only exercises record fields.

The demo is not an experiment

The action source is a transparent placeholder policy, so demo output goes to gitignored results/_demo/. It is not used for dissertation claims.

Setup

Setup by backend

--backend libero

Real ground-truth path. Build the state-only LIBERO env and add sklearn for the eval layer.

Add sklearn to libero14 venvnumpy<1.24 compatible
L14=~/.cache/evasion_tax-libero14/venv/bin/python
uv pip install --python "$L14" "scikit-learn==1.3.2"

--backend synthetic

Zero-setup path. It runs in the core .venv and emits the same record schema without a simulator.

Use this as the portable smoke path. It does not validate real LIBERO ground truth.

Variantspolicy/backend switches
PYTHONPATH=src:$LIB "$L14" scripts/demo_rollout.py --backend libero --policy random --seed 1
PYTHONPATH=src .venv/bin/python scripts/demo_rollout.py --backend synthetic
  • --steps N: rollout length, at least 4.
  • --seed S: pinned; same seed gives byte-identical records.
  • --policy {scripted,random}: transparent placeholder action source.
  • --backend {auto,libero,synthetic}: auto chooses LIBERO if importable.

Artifacts

What one run saves

run.json

Run protocol block with run_id, git_commit, hardware/env snapshot, config, seed, and hypothesis placeholders.

rollout_benign.json / rollout_attacked.json

A sequence of RolloutStep records with 7-DoF actions, privileged_state, instruction, trusted_goal, attacked flag, and suffix_ref.

actions_*.npy

Raw 7-DoF action arrays, so the same data can be re-analysed later.

attack_outcome.json

D2 success criterion. Records TargetActionSpec.reached_window and completion_step.

Small-sample behaviour is shown honestly

The demo n is small, so calibration splits are coarse and held-out FPR can overshoot the target. That is honest small-sample behaviour, not a bug.
